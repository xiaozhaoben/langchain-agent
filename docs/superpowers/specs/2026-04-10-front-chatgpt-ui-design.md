# front 前端仿 ChatGPT 聊天界面设计

## 背景

当前 `front` 模块采用原生 HTML、CSS、JavaScript 实现了一个单页聊天界面，并通过 `front/server.py` 调用后端 `/api/chat` SSE 接口完成流式回复。现有页面具备基础问答能力，但整体更接近“表单式问答页”，与用户期望的 ChatGPT 风格聊天工作台仍有明显差距。

本次改造目标是在不引入新前端框架、不修改后端接口协议的前提下，将 `front` 页面调整为更接近 ChatGPT 的深色聊天界面，同时保留当前单会话模型和已有流式输出逻辑。

## 目标

- 将页面改造成 ChatGPT 风格的深色布局，提升第一眼的产品感和聊天感。
- 保留当前单会话机制，不实现真实多会话列表。
- 保留现有 SSE 流式回复、建议问题点击填充、清空会话、会话缓存等能力。
- 修复当前页面中存在的中文乱码文案，统一为正常中文界面文本。
- 保证桌面端与移动端都能正常使用，重点优化输入区贴底和消息阅读体验。

## 非目标

- 不改动 `agent`、`rag`、`model` 等后端业务逻辑。
- 不新增 Vue、React 或构建工具，继续使用原生静态资源。
- 不实现真实的历史会话列表、会话切换、模型切换。
- 不改造 `/api/chat` 协议和流式响应格式。

## 方案选型

本次采用“高仿布局 + 轻量交互增强方案”。

### 备选方案

1. 只改视觉外观，不重构页面结构。
2. 改为 ChatGPT 风格布局，并增强欢迎态、建议问题、输入区贴底、消息流式体验。
3. 进一步模拟多会话列表和会话切换。

### 最终选择

选择方案 2。

原因如下：

- 与用户期望的“仿照 ChatGPT 聊天界面”最匹配。
- 不引入真实多会话复杂度，改造成本可控。
- 可以完全复用现有 SSE 接口与前端缓存逻辑。
- 对 `front` 模块是结构清晰的一次前端重构，不会牵动后端。

## 页面结构设计

页面改为桌面端双栏布局，主体结构如下：

1. 左侧边栏
2. 右侧主聊天区

### 左侧边栏

左侧边栏宽度固定，承担导航与快捷入口角色，内容从上到下如下：

- `新对话` 主按钮
- `清空会话` 次级按钮
- 推荐问题列表，作为快捷提问入口
- 项目名称与服务状态说明

边栏不展示真实历史会话列表，但在视觉层级、留白、按钮样式上尽量贴近 ChatGPT 桌面端体验。

### 主聊天区

主聊天区分为三个层级：

1. 顶部轻量标题栏
2. 中部消息滚动区
3. 底部固定输入区

顶部标题栏只展示产品名称和简短描述，不增加复杂操作，避免分散注意力。

中部消息区为主要阅读区域。当无消息时显示欢迎态；当有消息时显示连续消息流，并保持可滚动。

底部输入区固定贴在主聊天区底部，整体更接近聊天产品而不是表单页。

## 交互设计

### 欢迎态

无消息时在主聊天区中央显示欢迎内容，包括：

- 一句欢迎标题
- 一句简短提示语
- 3 到 4 个建议问题卡片

建议问题卡片点击后将内容填入输入框，并将焦点切回输入框。

### 消息流

消息流保留当前的用户消息和助手消息两种角色，但表现形式调整如下：

- 用户消息右对齐，采用较深但不过度突兀的胶囊样式
- 助手消息左对齐，气泡感减弱，强调阅读舒适度
- 角色标识简化为轻量标签或小头像区域
- 单条消息宽度限制在舒适阅读范围内，避免桌面端全文横向过宽

### 流式回复

继续沿用当前 `/api/chat` 的 SSE 流式响应逻辑：

- 用户发送消息后立即追加一条空助手消息
- 流式内容逐步填充到最后一条助手消息中
- 回复过程中自动滚动到底部
- 回复完成后更新状态提示

### 输入区

输入区调整为 ChatGPT 风格的底部输入容器，包含：

- 自适应高度文本框
- 发送按钮
- 轻量状态提示

交互规则保持如下：

- `Enter` 发送消息
- `Shift + Enter` 换行
- 发送中禁用输入框和发送按钮
- 清空会话后恢复欢迎态并重新聚焦输入框

## 视觉设计

### 风格方向

采用深色 ChatGPT 风格界面，整体原则如下：

- 页面主背景使用深灰色系
- 左侧边栏比主区域更深一层
- 强调文字可读性与信息层级
- 减少玻璃拟态、渐变和大面积高饱和装饰
- 保留少量产品语义文案，但不再大面积强调品牌色

### 色彩层级

计划建立统一的 CSS 变量体系，覆盖：

- 页面背景
- 侧栏背景
- 主内容背景
- 分隔线
- 主文本
- 次文本
- 用户消息背景
- 助手消息背景
- 按钮主色与悬停态
- 错误态颜色

### 响应式策略

桌面端采用左右分栏。

移动端改为上下结构：

- 左侧边栏收缩为顶部区域或较轻量的折叠块
- 主聊天区优先展示
- 输入区仍固定在底部，保证聊天体验连续

## 代码改造范围

### `front/index.html`

负责重构页面骨架，包括：

- 引入边栏容器
- 引入主聊天容器
- 调整欢迎态结构
- 调整消息模板结构
- 调整输入区结构

### `front/static/styles.css`

负责重写或大幅调整样式，包括：

- 深色主题变量
- 双栏布局
- 消息样式
- 欢迎态卡片
- 输入区贴底布局
- 桌面端和移动端响应式样式

### `front/static/app.js`

负责适配新 DOM 结构和交互逻辑，包括：

- 更新元素选择器
- 适配欢迎态与推荐问题渲染
- 适配新消息结构
- 保留现有会话缓存逻辑
- 保留现有 SSE 流式输出逻辑
- 调整状态提示与滚动行为

## 数据流与状态设计

本次不新增复杂状态模型，继续使用当前轻量状态结构：

- `state.messages`：消息列表
- `state.isSending`：请求发送状态
- `typingState.queue`：流式文本队列
- `sessionStorage`：会话缓存

数据流保持如下：

1. 用户输入消息
2. 前端追加用户消息和占位助手消息
3. 前端发起 `/api/chat` 请求
4. SSE 数据按块到达并写入占位助手消息
5. 回复结束后保存缓存并更新状态

## 错误处理

保留现有错误处理原则，并在界面上更自然地呈现：

- 空消息禁止发送
- 接口异常时，在最后一条助手消息中展示错误提示
- 状态区同步显示错误状态
- 网络异常或浏览器不支持流式响应时，给出明确中文提示

## 测试与验证

实现后需要完成以下验证：

1. 页面可正常加载，且无中文乱码文案。
2. 无消息时显示欢迎态和建议问题卡片。
3. 点击建议问题可将文本写入输入框。
4. 发送消息后，用户消息和助手占位消息正常渲染。
5. SSE 流式回复可持续更新最后一条助手消息。
6. 回复结束后消息缓存正常写入 `sessionStorage`。
7. 点击 `清空会话` 后恢复欢迎态。
8. 桌面端布局为左侧栏 + 主聊天区。
9. 移动端布局不会导致输入区遮挡消息内容。

## 风险与控制

### 风险 1：原始 DOM 结构变化较大，导致现有事件绑定失效

控制方式：在改造前明确新旧结构映射关系，保留核心 DOM id，减少脚本断裂风险。

### 风险 2：输入区贴底后，消息区滚动和视口高度处理不稳定

控制方式：采用明确的主聊天区高度和内部滚动容器设计，避免继续依赖整个页面滚动。

### 风险 3：深色主题下消息层级不清楚

控制方式：通过统一变量体系控制文本、背景、边框对比度，并优先保障阅读性。

## 实施顺序

1. 重构 `index.html` 页面骨架
2. 重写 `styles.css` 建立深色布局与组件样式
3. 调整 `app.js` 以适配新结构
4. 启动 `front/server.py` 进行手工验证
5. 修正布局与交互问题

## 交付结果

完成后，`front` 页面应从当前的浅色卡片式问答页，升级为更接近 ChatGPT 的深色聊天工作台，在不改变后端接口与单会话模式的前提下，显著提升视觉一致性和聊天体验。
